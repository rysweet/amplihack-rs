## Test Organization Patterns

### Pattern 1: Feature-Based Organization

Organize tests by user-facing features rather than technical layers.

**Structure:**

```
e2e/
├── user-authentication/
│   ├── registration.spec.ts
│   ├── login.spec.ts
│   └── password-reset.spec.ts
├── product-catalog/
│   ├── product-search.spec.ts
│   ├── product-detail.spec.ts
│   └── product-comparison.spec.ts
└── checkout/
    ├── cart-management.spec.ts
    ├── shipping.spec.ts
    └── payment.spec.ts
```

**Benefits:**

- Easy to find tests for specific features
- Clear responsibility boundaries
- Facilitates feature-level test parallelization

**Example:**

```typescript
// e2e/user-authentication/registration.spec.ts
test.describe('User Registration', () => {
  test('happy path: valid registration', async ({ page }) => { ... });
  test('edge case: duplicate email', async ({ page }) => { ... });
  test('security: prevents SQL injection', async ({ page }) => { ... });
});
```

### Pattern 2: Journey-Based Organization

Organize tests by complete user journeys spanning multiple features.

**Structure:**

```
e2e/
├── journeys/
│   ├── new-user-first-purchase.spec.ts
│   ├── returning-user-reorder.spec.ts
│   └── guest-checkout.spec.ts
├── features/
│   ├── search.spec.ts
│   ├── cart.spec.ts
│   └── checkout.spec.ts
```

**Benefits:**

- Tests reflect real user behavior
- Catches integration issues between features
- Validates complete business workflows

**Example:**

```typescript
// e2e/journeys/new-user-first-purchase.spec.ts
test("new user completes first purchase", async ({ page }) => {
  // Registration
  await registerUser(page, "newuser@example.com", "Password123!");

  // Product search
  await page.goto("/products");
  await page.getByRole("searchbox").fill("laptop");
  await page.getByRole("button", { name: /search/i }).click();

  // Add to cart
  await page.getByRole("link", { name: /first result/i }).click();
  await page.getByRole("button", { name: /add to cart/i }).click();

  // Checkout
  await page.goto("/checkout");
  await fillShippingInfo(page);
  await fillPaymentInfo(page);
  await page.getByRole("button", { name: /place order/i }).click();

  // Verification
  await expect(page).toHaveURL(/\/orders\/\d+/);
  await expect(page.getByText(/order confirmed/i)).toBeVisible();
});
```

### Pattern 3: Category-Based Organization

The default organization generated by the skill.

**Structure:**

```
e2e/
├── happy-path/
├── edge-cases/
├── error-handling/
├── performance/
├── security/
├── accessibility/
└── integration/
```

**Benefits:**

- Clear test type separation
- Easy to run specific test categories
- Aligns with testing best practices

## Locator Strategy Patterns

### Pattern 1: Role-Based Locators (Preferred)

Use ARIA roles for the most resilient locators.

**Pattern:**

```typescript
// Prefer role-based locators
await page.getByRole("button", { name: /submit/i });
await page.getByRole("textbox", { name: /email/i });
await page.getByRole("link", { name: /home/i });
await page.getByRole("heading", { name: /welcome/i });
```

**Benefits:**

- Resilient to UI changes
- Enforces accessibility best practices
- Works across different implementations

**Anti-Pattern:**

```typescript
// Avoid CSS selectors
await page.click(".btn-primary");
await page.fill("#email-input");
```

### Pattern 2: Test ID Strategy

Use test IDs for elements without semantic roles.

**HTML:**

```html
<div data-testid="shopping-cart-count">3</div>
<button data-testid="checkout-button">Checkout</button>
```

**Test:**

```typescript
const cartCount = await page.getByTestId("shopping-cart-count").textContent();
await page.getByTestId("checkout-button").click();
```

**Configuration:**

```typescript
// playwright.config.ts
export default defineConfig({
  use: {
    testIdAttribute: "data-testid",
  },
});
```

**Benefits:**

- Explicit test hooks
- Stable across refactoring
- Clear intent

### Pattern 3: Composite Locator Strategy

Combine multiple strategies for complex scenarios.

**Pattern:**

```typescript
// Find button within specific section
await page
  .getByRole("region", { name: /shipping/i })
  .getByRole("button", { name: /continue/i })
  .click();

// Find specific list item
await page
  .getByRole("list", { name: /products/i })
  .getByRole("listitem")
  .filter({ hasText: /laptop/i })
  .click();
```

**Benefits:**

- Precise targeting
- Avoids ambiguity
- Self-documenting intent

## Data Management Patterns

### Pattern 1: Fixture-Based Test Data

Use JSON fixtures for predictable test data.

**Fixture:**

```json
// e2e/fixtures/users.json
[
  {
    "id": 1,
    "email": "test@example.com",
    "password": "Test123!", // pragma: allowlist secret
    "role": "customer"
  },
  {
    "id": 2,
    "email": "admin@example.com",
    "password": "Admin123!", // pragma: allowlist secret
    "role": "admin"
  }
]
```

**Usage:**

```typescript
import users from "../fixtures/users.json";

test("customer can login", async ({ page }) => {
  const customer = users.find((u) => u.role === "customer")!;
  await login(page, customer.email, customer.password);
  await expect(page).toHaveURL("/dashboard");
});
```

**Benefits:**

- Predictable data
- Easy to maintain
- Version controlled

### Pattern 2: Factory Functions

Generate test data programmatically for variation.

**Factory:**

```typescript
// e2e/test-helpers/factories.ts
export function createUser(overrides = {}) {
  return {
    email: `test-${Date.now()}@example.com`,
    password: "Test123!", // pragma: allowlist secret
    name: "Test User",
    ...overrides,
  };
}

export function createProduct(overrides = {}) {
  return {
    name: `Product ${Date.now()}`,
    price: 99.99,
    stock: 10,
    ...overrides,
  };
}
```

**Usage:**

```typescript
test("user can register", async ({ page }) => {
  const user = createUser({ name: "Alice" });
  await register(page, user);
  await expect(page.getByText(`Welcome, ${user.name}`)).toBeVisible();
});
```

**Benefits:**

- Unique data per test run
- Prevents data conflicts
- Flexible customization

### Pattern 3: Setup/Teardown Data Management

Use `beforeEach`/`afterEach` for consistent test state.

**Pattern:**

```typescript
test.describe("Product Management", () => {
  let testProduct: Product;

  test.beforeEach(async ({ page }) => {
    // Setup: Create fresh product
    testProduct = await createProductViaAPI({
      name: "Test Product",
      price: 99.99,
    });
  });

  test.afterEach(async () => {
    // Teardown: Delete test product
    await deleteProductViaAPI(testProduct.id);
  });

  test("user can view product", async ({ page }) => {
    await page.goto(`/products/${testProduct.id}`);
    await expect(page.getByText(testProduct.name)).toBeVisible();
  });
});
```

**Benefits:**

- Isolated test state
- No data pollution
- Predictable test environment

## Integration Patterns

### Pattern 1: API-First Data Setup

Use API calls to set up test data, UI to verify.

**Pattern:**

```typescript
import { request } from "@playwright/test";

test("order appears in order history", async ({ page }) => {
  // Setup via API (fast)
  const apiContext = await request.newContext();
  const orderResponse = await apiContext.post("/api/orders", {
    data: {
      productId: 123,
      quantity: 1,
      userId: 456,
    },
  });
  const order = await orderResponse.json();

  // Verify via UI (user-facing)
  await page.goto("/orders");
  await expect(page.getByText(`Order #${order.id}`)).toBeVisible();
});
```

**Benefits:**

- Faster test execution
- Focuses UI tests on UI
- Separates concerns

### Pattern 2: Database Seeding

Seed database directly for complex test scenarios.

**Setup:**

```typescript
// e2e/test-helpers/db.ts
import { Pool } from "pg";

const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
});

export async function seedDatabase(data: any) {
  await pool.query("BEGIN");
  try {
    // Insert test data
    await pool.query("INSERT INTO users VALUES ($1, $2)", [data.userId, data.email]);
    await pool.query("INSERT INTO orders VALUES ($1, $2)", [data.orderId, data.userId]);
    await pool.query("COMMIT");
  } catch (error) {
    await pool.query("ROLLBACK");
    throw error;
  }
}

export async function cleanDatabase() {
  await pool.query("TRUNCATE users, orders CASCADE");
}
```

**Usage:**

```typescript
import { seedDatabase, cleanDatabase } from "../test-helpers/db";

test.describe("Order History", () => {
  test.beforeEach(async () => {
    await cleanDatabase();
    await seedDatabase({
      userId: 1,
      email: "test@example.com",
      orderId: 100,
    });
  });

  test("displays seeded order", async ({ page }) => {
    await login(page, "test@example.com", "Test123!");
    await page.goto("/orders");
    await expect(page.getByText("Order #100")).toBeVisible();
  });
});
```

**Benefits:**

- Complex data setup
- Full control over state
- Fast execution

### Pattern 3: Stubbed External Services

Stub external APIs for isolated testing.

**Pattern:**

```typescript
test("handles payment gateway timeout", async ({ page, context }) => {
  // Stub external payment API
  await context.route("**/api/payments/process", (route) => {
    setTimeout(() => {
      route.abort("timedout");
    }, 5000);
  });

  await page.goto("/checkout");
  await fillPaymentInfo(page);
  await page.getByRole("button", { name: /pay now/i }).click();

  await expect(page.getByText(/payment timeout/i)).toBeVisible();
  await expect(page.getByRole("button", { name: /retry/i })).toBeVisible();
});
```

**Benefits:**

- Predictable external behavior
- Test error scenarios
- No external dependencies

## Customization Patterns

### Pattern 1: Custom Helper Modules

Create reusable helper functions for common operations.

**Structure:**

```
e2e/test-helpers/
├── auth.ts        # Authentication operations
├── navigation.ts  # Navigation helpers
├── forms.ts       # Form filling
├── assertions.ts  # Custom assertions
└── api.ts         # API operations
```

**Example:**

```typescript
// e2e/test-helpers/forms.ts
import { Page } from "@playwright/test";

export async function fillLoginForm(page: Page, email: string, password: string) {
  await page.getByRole("textbox", { name: /email/i }).fill(email);
  await page.getByRole("textbox", { name: /password/i }).fill(password);
}

export async function fillShippingForm(page: Page, address: Address) {
  await page.getByRole("textbox", { name: /street/i }).fill(address.street);
  await page.getByRole("textbox", { name: /city/i }).fill(address.city);
  await page.getByRole("textbox", { name: /zip/i }).fill(address.zip);
}
```

**Usage:**

```typescript
import { fillLoginForm } from "../test-helpers/forms";

test("user can login", async ({ page }) => {
  await page.goto("/login");
  await fillLoginForm(page, "test@example.com", "Test123!");
  await page.getByRole("button", { name: /sign in/i }).click();
  await expect(page).toHaveURL("/dashboard");
});
```

**Benefits:**

- DRY principle
- Consistent behavior
- Easy maintenance

### Pattern 2: Custom Playwright Fixtures

Create reusable test context with fixtures.

**Fixture:**

```typescript
// e2e/fixtures.ts
import { test as base } from "@playwright/test";

type Fixtures = {
  authenticatedPage: Page;
  adminPage: Page;
};

export const test = base.extend<Fixtures>({
  authenticatedPage: async ({ page }, use) => {
    await login(page, "test@example.com", "Test123!");
    await use(page);
  },

  adminPage: async ({ page }, use) => {
    await login(page, "admin@example.com", "Admin123!");
    await use(page);
  },
});
```

**Usage:**

```typescript
import { test } from "../fixtures";

test("authenticated user sees dashboard", async ({ authenticatedPage }) => {
  await authenticatedPage.goto("/dashboard");
  await expect(authenticatedPage.getByText(/welcome/i)).toBeVisible();
});

test("admin can access admin panel", async ({ adminPage }) => {
  await adminPage.goto("/admin");
  await expect(adminPage.getByText(/admin panel/i)).toBeVisible();
});
```

**Benefits:**

- Reusable test contexts
- Automatic setup/teardown
- Type-safe fixtures

### Pattern 3: Page Object Model (POM)

Encapsulate page interactions in classes.

**Page Object:**

```typescript
// e2e/pages/LoginPage.ts
import { Page, Locator } from "@playwright/test";

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitButton: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.getByRole("textbox", { name: /email/i });
    this.passwordInput = page.getByRole("textbox", { name: /password/i });
    this.submitButton = page.getByRole("button", { name: /sign in/i });
  }

  async goto() {
    await this.page.goto("/login");
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }
}
```

**Usage:**

```typescript
import { LoginPage } from "../pages/LoginPage";

test("user can login", async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login("test@example.com", "Test123!");
  await expect(page).toHaveURL("/dashboard");
});
```

**Benefits:**

- Encapsulation
- Maintainable locators
- Reusable page logic

**When to Use:**

- Large test suites (100+ tests)
- Complex page interactions
- Multiple tests using same page

## CI/CD Patterns

### Pattern 1: Progressive Test Execution

Run fast tests first, slow tests later.

**GitHub Actions:**

```yaml
# .github/workflows/e2e.yml
jobs:
  smoke-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npm run test:e2e:smoke # Fast, critical tests

  full-suite:
    needs: smoke-tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npm run test:e2e # Full test suite
```

**Package.json:**

```json
{
  "scripts": {
    "test:e2e:smoke": "playwright test e2e/happy-path --grep-invert @slow",
    "test:e2e": "playwright test"
  }
}
```

**Benefits:**

- Fast feedback
- Fail fast on critical issues
- Efficient CI resource usage

### Pattern 2: Sharded Test Execution

Split tests across multiple CI jobs (with care).

**Note**: This pattern requires careful data isolation as the skill enforces `workers=1`.

**GitHub Actions:**

```yaml
jobs:
  e2e:
    strategy:
      matrix:
        shard: [1, 2, 3, 4]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test --shard=${{ matrix.shard }}/4
        env:
          # CRITICAL: Each shard needs isolated database
          DATABASE_URL: postgresql://test:test@localhost:5432/testdb_${{ matrix.shard }} # pragma: allowlist secret
```

**Benefits:**

- Faster CI execution
- Parallel test runs

**Caution:**

- Requires isolated databases per shard
- More complex setup
- Only use if test suite is very large (>1000 tests)

### Pattern 3: Test Artifacts on Failure

Collect screenshots, videos, traces on failure.

**GitHub Actions:**

```yaml
jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npm run test:e2e

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v3
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30

      - name: Upload failure artifacts
        if: failure()
        uses: actions/upload-artifact@v3
        with:
          name: test-failures
          path: |
            e2e/test-results/
            playwright-report/
          retention-days: 7
```

**Playwright Config:**

```typescript
export default defineConfig({
  use: {
    screenshot: "only-on-failure",
    video: "retain-on-failure",
    trace: "on-first-retry",
  },
});
```

**Benefits:**

- Debug failures easily
- Visual failure evidence
- Trace replay capability

## Anti-Patterns to Avoid

### Anti-Pattern 1: Parallel Test Execution (workers > 1)

**DO NOT DO THIS:**

```typescript
// playwright.config.ts
export default defineConfig({
  workers: 4, // WRONG - causes data races
});
```

**Why it fails:**

- Tests share database
- Data race conditions
- Flaky tests
- Unpredictable failures

**Correct approach:**

```typescript
// playwright.config.ts
export default defineConfig({
  workers: 1, // CORRECT - enforced by skill
  fullyParallel: false,
});
```

### Anti-Pattern 2: Hardcoded Waits

**DO NOT DO THIS:**

```typescript
await page.click("#submit");
await page.waitForTimeout(3000); // WRONG - arbitrary wait
```

**Why it fails:**

- Flaky tests
- Slow execution
- Unpredictable timing

**Correct approach:**

```typescript
await page.click("#submit");
await page.waitForLoadState("networkidle"); // CORRECT - explicit condition
await expect(page.getByText(/success/i)).toBeVisible(); // Auto-waits
```

### Anti-Pattern 3: CSS Selector Over-Reliance

**DO NOT DO THIS:**

```typescript
await page.click(".btn.btn-primary.submit-btn"); // WRONG - brittle
await page.fill("#user-email-input-field"); // WRONG - implementation detail
```

**Why it fails:**

- Breaks on UI refactoring
- Fragile locators
- Hard to maintain

**Correct approach:**

```typescript
await page.getByRole("button", { name: /submit/i }).click(); // CORRECT - semantic
await page.getByRole("textbox", { name: /email/i }).fill("test@example.com"); // CORRECT - accessible
```
