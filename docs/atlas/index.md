## Rendering

Atlas diagrams are regenerated by the **Code Atlas** GitHub Actions workflow
(`.github/workflows/atlas.yml`) on every push to `main` and on manual
`workflow_dispatch`. The workflow installs Graphviz, renders the Graphviz DOT
source for each layer to SVG, and uploads the entire `docs/atlas/` tree as the
`code-atlas` build artifact (90-day retention).

### Diagram source formats

The atlas uses two diagram source formats. A single layer may carry **both** â€”
most layers ship a Graphviz DOT source *and* a Mermaid source â€” so the formats
are not mutually exclusive:

| Format | Source file | Rendered by | Output |
| ------ | ----------- | ----------- | ------ |
| Graphviz DOT | `docs/atlas/<layer>/<layer>.dot` | `dot -Tsvg` in the atlas workflow | `<layer>-dot.svg` |
| Mermaid | `docs/atlas/<layer>/*.mmd` | Mermaid tooling (committed `*-mermaid.svg` alongside source) | `*-mermaid.svg` |

Only the **Graphviz DOT** format is rendered by the atlas workflow; Mermaid
`*-mermaid.svg` files are committed by contributors. The render loop therefore
keys off the presence of a `<layer>.dot` file, not off whether the layer also
has Mermaid sources.

### DOT-rendered layers

The "Render DOT diagrams" step iterates **only** the layers that ship a
`<layer>.dot` file. These seven layers each have a `.dot` source (and also carry
a companion `<layer>.mmd` Mermaid source), and are rendered by Graphviz:

- `repo-surface`
- `ast-lsp-bindings`
- `compile-deps`
- `runtime-topology`
- `api-contracts`
- `data-flow`
- `service-components`

Each iteration runs:

```bash
dot -Tsvg "docs/atlas/$layer/$layer.dot" \
    -o "docs/atlas/$layer/${layer}-dot.svg"
```

### Mermaid-only layers

`user-journeys` is the only **Mermaid-only** layer. It has no `<layer>.dot`
file â€” its diagrams live in `docs/atlas/user-journeys/*.mmd` (`user-journeys.mmd`
plus the per-journey `journey-install.mmd`, `journey-recipe-run.mmd`,
`journey-hooks.mmd`) with committed `*-mermaid.svg` renders. Because Mermaid-only
layers are **excluded**
from the DOT render loop, the workflow never invokes `dot` against a missing
`user-journeys.dot`, and the "Render DOT diagrams" step completes successfully.

> **Note:** When adding a new layer, include it in the DOT render loop **only**
> if you commit a matching `docs/atlas/<layer>/<layer>.dot` file. Mermaid-only
> layers must be left out of the loop, otherwise `dot` exits with code 2 on the
> missing source file and fails the workflow.
